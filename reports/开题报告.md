# 开题报告：基于脏数据基准的 LLM Data Agent 评估框架

> **团队人数**：5 人
> **项目方向**：Agentic 数据科学家评估平台

---

## 1. 问题定义

### 1.1 研究背景与动机

大语言模型（LLM）驱动的智能体（Agent）正快速渗透数据科学工作流。以 ChatGPT Code Interpreter 为代表的工具已经展示了 LLM 自动完成数据探索、清洗和建模的可能性。然而，当前对 Data Agent 的评估大多局限于"能否跑通任务"的定性展示，缺乏一套系统性的定量评测框架来回答以下核心问题：

- **检测能力**：Agent 能否识别数据中的缺失值、标签噪声、数据泄漏等常见缺陷？
- **修复质量**：Agent 的数据清洗操作是否真正提升了下游模型性能？
- **过程规范性**：Agent 是否遵循了数据科学的最佳实践（如先划分训练/测试集再做特征工程）？

本项目旨在构建一套多层次的 Data Agent 评测基准（benchmark），并基于 LangChain/LangGraph + DeepSeek 搭建可控的 Agent 系统，通过消融实验系统评估不同 Agent 设计策略对数据处理质量的影响。

### 1.2 任务形式化

定义评测任务为一个四元组 $(D', T, G, f)$：

- $D'$：含有已知缺陷的脏数据集
- $T$：自然语言任务描述（如"请清洗该数据集并训练一个二分类模型"）
- $G$：Ground truth，包括已知缺陷列表 $G_{\text{defect}}$ 和干净数据上的模型性能上界 $G_{\text{perf}}$
- $f$：Agent 系统

Agent 接收 $(D', T)$，输出数据处理管线 $P$ 和最终模型 $M$。评估体系衡量 $f$ 在三个维度上的表现：

$$\text{Score}(f) = \alpha \cdot \text{Detection}(P, G_{\text{defect}}) + \beta \cdot \text{Recovery}(M, G_{\text{perf}}) + \gamma \cdot \text{Process}(P)$$

其中 Detection 衡量缺陷检出能力，Recovery 衡量性能恢复程度，Process 衡量过程规范性。

## 2. 数据来源证明与初步探勘

### 2.1 三层数据集设计

本项目采用合成、半合成、真实数据三层结构，兼顾评测精度和现实代表性：

**第一层：合成数据（精确评测）**

使用 `sklearn.datasets.make_classification()` 生成基础数据集，随后注入六类已知缺陷：

| 缺陷类型 | 注入方式 | Ground Truth |
|---------|---------|-------------|
| 数据泄漏 | 将目标变量的线性变换混入特征 | 泄漏列的索引 |
| 缺失值 | 按 MCAR/MAR/MNAR 机制随机置空 | 被置空的坐标及原始值 |
| 标签噪声 | 随机翻转 5%/10%/20% 标签 | 被翻转的样本索引 |
| 近似重复 | 复制样本并添加微小高斯扰动 | 重复样本对 |
| 类别不平衡 | 对少数类进行欠采样 | 原始类别分布 |
| 格式不一致 | 同一列混入不同单位或编码方式 | 异常格式的行索引 |

每个合成数据集规模约 10,000 行、20-50 特征，计划生成 2-3 个不同难度梯度的版本。

**第二层：CleanML Benchmark（半合成评测）**

采用 CleanML（Li et al., NeurIPS 2021）提供的脏/净配对数据集。该 benchmark 包含多个真实数据集在不同脏化条件下的标准化版本，自带 ground truth 性能上界，可直接计算 Recovery Rate。计划从中选取 2-3 个表格分类任务（如 Marketing、EEG 等）。

- 数据来源：CleanML GitHub 仓库（MIT License，已确认可访问）

**第三层：Kaggle 真实数据集（端到端评测）**

选取 1-2 个本身存在数据质量问题的 Kaggle 竞赛数据集，与团队成员的人工处理结果进行对比。候选：

- Home Credit Default Risk（信贷风控，特征 120+，存在大量缺失值和类别编码问题）
- IEEE-CIS Fraud Detection（欺诈检测，存在高维稀疏特征和类别不平衡）

### 2.2 初步挑战

- **缺陷梯度设计**：需合理设置不同脏化程度（轻度/中度/重度），避免任务过简或过难
- **跨数据集一致性**：不同数据集的缺陷类型和严重程度差异较大，需设计统一的评分框架
- **人类 baseline 的主观性**：第三层数据集的人工处理结果依赖团队成员水平，需记录处理过程以增强可复现性

## 3. 初步方法与技术栈

### 3.1 基线方法

- **下界（No-clean baseline）**：直接在脏数据上训练模型（XGBoost / LightGBM），不做任何数据清洗
- **规则基线（Rule-based baseline）**：使用 `ydata-profiling`（原 pandas-profiling）自动生成数据质量报告，按其建议执行固定清洗规则（如均值填充缺失值、删除重复行）
- **人类基线（Human baseline）**：团队成员手动执行完整的数据清洗和特征工程流程（仅限第三层数据集）

### 3.2 核心方案：LangGraph Data Agent

使用 LangGraph 构建具有状态管理能力的 Data Agent，核心架构：

```
用户任务描述
    ↓
[Planner Node] → 生成数据处理计划
    ↓
[Executor Node] → 调用工具执行（pandas, sklearn 等）
    ↓
[Reviewer Node] → 检查执行结果，决定是否修正（Reflection）
    ↓
[Reporter Node] → 输出最终清洗后数据集 + 模型 + 分析报告
```

底层 LLM 选用 DeepSeek（通过 API 调用），其在代码生成和数据分析任务上有较好的性价比。

### 3.3 消融变量

| 消融维度 | 变体 | 目的 |
|---------|------|------|
| Reflection 机制 | 有 / 无 Reviewer Node | 验证自我检查是否提升清洗质量 |
| 工具调用 | 有 / 无 pandas/sklearn 工具 | 验证工具使用 vs 纯代码生成的差异 |
| Prompt 策略 | Zero-shot / Few-shot / CoT | 验证不同提示策略对 Agent 行为的影响 |
| 规划能力 | 有 / 无 Planner Node | 验证先规划再执行是否优于直接执行 |

### 3.4 技术栈

- **Agent 框架**：LangGraph (Python)
- **LLM**：DeepSeek API
- **数据处理**：pandas, numpy, scikit-learn
- **下游验证模型**：XGBoost, LightGBM（非研究对象，仅作为固定的评估工具衡量数据清洗效果）
- **数据质量分析**：ydata-profiling
- **实验管理**：MLflow 或 Weights & Biases（记录每次实验配置和结果）

## 4. 实验设计与评估指标

### 4.1 评估指标体系

**A. 缺陷检测指标（第一层数据集）**

- 检出率（Recall）：Agent 正确识别的缺陷数 / 已知注入缺陷总数
- 精确率（Precision）：Agent 正确识别的缺陷数 / Agent 报告的缺陷总数
- 分缺陷类型统计：分别计算各类缺陷（泄漏/缺失/噪声/重复/不平衡/格式）的检出情况

**B. 清洗效果指标（第一、二层数据集）**

为确保公平对比，所有条件下的模型均在**同一个固定的干净测试集**上评估。具体协议：

1. 从干净数据中预先划出 20% 作为固定测试集 $D_{\text{test}}$，全程不参与清洗和训练
2. 剩余 80% 注入缺陷形成脏训练集 $D'_{\text{train}}$
3. 各条件（不清洗 / Agent 清洗 / 干净数据）分别在各自的训练集上训练 XGBoost，但统一在 $D_{\text{test}}$ 上计算 AUC-ROC

$$\text{Recovery Rate} = \frac{\text{AUC}_{\text{Agent}} - \text{AUC}_{\text{dirty}}}{\text{AUC}_{\text{clean}} - \text{AUC}_{\text{dirty}}}$$

其中三个 AUC 均来自同一个 $D_{\text{test}}$。Recovery Rate = 1 表示完全恢复干净数据的性能，> 1 表示 Agent 的处理甚至优于原始"干净"数据。

**C. 端到端对比指标（第三层数据集）**

- 最终模型 AUC-ROC / F1-score
- 与人类 baseline 的相对差距

**D. 过程质量 Checklist（所有数据集）**

对 Agent 的处理过程进行逐项检查（0/1 评分）：

- [ ] 是否进行了数据探索性分析（EDA）
- [ ] 是否在划分训练/测试集之后才做特征工程
- [ ] 是否检测并处理了数据泄漏风险
- [ ] 是否识别并处理了缺失值
- [ ] 是否识别并处理了类别不平衡
- [ ] 是否对异常值/重复值进行了检查

### 4.2 消融实验设计

在第一层和第二层数据集上，对 3.3 节中定义的四个消融维度进行控制变量实验。每个配置运行 3 次取均值，以减少 LLM 输出的随机性影响。总计约 $2^4 \times 3 = 48$ 次实验运行。

## 5. 风险清单与排期

### 5.1 风险评估

| 风险 | 影响 | 预案 |
|------|------|------|
| DeepSeek API 不稳定或响应延迟高 | 实验效率低，结果不可复现 | 引入本地部署的小参数量开源模型（如 Qwen2.5-Coder-7B）作为补充对比 |
| Agent 在所有配置下表现均很差 | 消融实验缺乏区分度 | 降低任务难度：从全自动改为半自动模式（Agent 提出建议，人工确认执行），同时分析失败原因本身也是有价值的研究发现 |

### 5.2 里程碑排期

| 时间 | 里程碑 | 主要产出 |
|------|-------|---------|
| 第 8 周 | 开题提交 | 开题报告 + GitHub 仓库初始化 |
| 第 9-10 周 | 数据与基线 | 三层数据集构建完成；No-clean / Rule-based baseline 跑通 |
| 第 10-11 周 | Agent 开发 | LangGraph Agent 核心框架搭建完成，单数据集跑通 |
| 第 12 周 | 中期检查 | 中期报告：Agent 在部分数据集上的初步结果 + 消融实验进展 |
| 第 13-15 周 | 全量实验 | 完成所有消融实验 + 人类 baseline 对比 |
| 第 16 周 | 最终答辩 | 完整实验结果分析 + 最终报告 + 答辩展示 |
